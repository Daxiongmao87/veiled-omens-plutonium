# Plutonium Schema Notes

This directory contains working schema/reference notes for creating Veiled Omens content in Plutonium/5etools format.

These are practical authoring references, not strict JSON Schema validator files. They were derived from a Foundry VTT JournalEntry export named `Plutonium Schemas`, which contained Plutonium content-type notes and live examples.

Use these notes to choose the correct top-level array and `__prop` value when adding content to this repository.

## Main Reference

- `plutonium-content-types.md` - compact reference for supported Plutonium content types, key fields, and authoring notes.
- `../docs/5etools-homebrew-conventions.md` - source-conventions mapping for TheGiddyLimit/homebrew format, directory names, and Foundry asset transplants.

## Content File Naming

- File paths follow source material/package identity.
- Use `collection/` when one source material package spans multiple content types.
- Use type-specific directories when the source package is a single-type package or a true type-specific collection.
- Keep source IDs tied to source material/package identity, not individual classes, subclasses, species, spells, items, or features.
- Keep author identity in JSON metadata (`_meta.sources[].authors`, adventure/book `author`) unless the upstream package filename convention requires the author segment.

## Important Distinction

Plutonium/5etools homebrew JSON wraps content in top-level arrays named after the content type:

```json
{
  "_meta": {
    "sources": [
      {
        "json": "VeiledOmens",
        "abbreviation": "VO",
        "full": "Veiled Omens Campaign Setting",
        "authors": ["Patrick Richardson"],
        "version": "1.0.0-foundry-mechanics"
      }
    ]
  },
  "race": [],
  "class": [],
  "classFeature": [],
  "subclass": [],
  "subclassFeature": [],
  "spell": [],
  "item": []
}
```

Individual entries often include `__prop`, but the top-level array name is still what determines where the entry belongs in a homebrew file.

## Validation Note

Before relying on any schema note, compare against one live official Plutonium entry of the same type. Some fields are Plutonium import fields, some are Foundry dnd5e-system fields, and some are display/filter metadata generated by Plutonium at runtime.

Plutonium's bundled 5etools source data uses string refs for normal class/subclass feature references, object refs for class-feature metadata such as subclass choice rows, and string entries for class tool-proficiency grants. Its side-data uses `advancement` rows for non-item values such as `ScaleValue`; it does not source-author `ItemGrant` rows for class, subclass, or race feature grants. Feature-grant `ItemGrant` rows are generated during actor import from race entries and class/subclass feature references. Subclass `subclassFeatures` arrays start with a subclass-named header feature; Plutonium imports that first feature into the subclass item and ignores it as a separate actor feature. Same-level mechanical subclass features are referenced inside that header with `refSubclassFeature`, and later-level features follow the header as sibling `subclassFeatures`.
